# A1.3.1 Role of operating systems 
An operating system manages **computer hardware** and 
**software resources**, and provide *common services for computer programs*. 

Its an abstraction layer between the user and the computer hardware. 

Resource management responsibilities:
1. CPU management: Allocation of CPU time to different processes. It creates, schedules, suspends and terminates processes and threads. It also implements context-switching an enforces privilege seperation. 

2. Memory management: Operating systems manage the allocation is de-allocation of memeory to applications, and manage virtual memory to etend the capabilities of harware

3. I/O and device management: Manages device drivers and **Direct memory access(DMA)**. 

4. File system and secondary-storage management: Implements naming, directories, metadata, journeling, and caching. 

5. Security and access control: Authenticates principles, enforeces disretionar /mandatority direct access control models. 

6. User interface subsystem: Provides command-line shells, window servers of system libraries so that application UI code dosent not need to address the GPUs directly.

## OS scheduling: A1.3.3 Approaches for sceduling

### Multilevel queue scedulign 

- Paritions the ready queue into several seperate queues, each with either own sceduling algorithm and priority level. 

Processes are permentaly assigned to one of these queus based on ceratin *charactersitics*, such as **process type, priority** or **memory requirements**. 

Each queue may use different sceduling algorithms, and the queues themselves are sceduled in a specific order, oftne based on priority. 

- Higher-priority queues are given more CPU time compared to lower-priority queues. 


Advantages:
1) Flexiblity in handling different types of processes 
2) Prioritizes critical and interactive processes, improving responsivness for imporant tasks
3) Different scheduling algorithms can be tailored to the needs of eahc queue 

Disadvantages: 
1) Starvation of lower-priority proceeses if higher-priority queues are frequently occupied 
2) Complex implementaiton and management 
3) Processes are permenantly assigned to quese which might not be opitmal if their behavior changes over time

## Polling and Interrupt handling 

https://www.youtube.com/watch?v=FzqlPDMy6Bk

Devices in a computer sends information to the CPU

- Polling: The repetitive monitoring/checking done by the CPU on a status register or memory location to see if anything action has occured. 

- Interupt handling: the external device assets a signal that asynchronously diverts the CPU to a service routine when the event occurs

### Interupt 
The device sends a asynchronous signal to the CPU that interupts the CPU's task at that point. 

### Polling 
(sortof opposite to interupt)
The device can't interupt the CPU directly, it can only reply to requests form the CPU

The CPU will continously be sending signals towards the device to check the status. 

This adds latency: an action done by the device is not always run at the same moment that the CPU sends a signal. There may be a time delay of a few milliseconds. 

More latency than interupt system

USB pheripherals use polling. 

### Comparison

| Criteria | Polling | Interrupt handling |
|----------|----------|----------|
|Event frequency|Suited for events that take place at high-rates or continous streams.|Best for sparse events so that the CPU does not have to interupt actions at a higher frequency|
|Overhead|The repetitive signal genertion and a bus transaction scales the overhead. |There is no overhead when no signal is geneated from a device. However the context switching during each inturrupt is evened out when no interuppt is geneated. |
|Power consumption| Continous polling prevents deep sleep states of device, which is undesirable on battery-powered devices|Allows the core to remain in low-power C-states untill the interupt in fired.|

#### Preference of polling vs interrupt 
Polling:
- Event rate is hige enought that most polls suceed (~ > 50% hit ratio)
- You need a strictly bounded latency shorter than the platforms worst-case interrupt latency 
- The CPU would be awake 

Interrupt handling: 
- Events are bursty or sparse
- Battery life or thermal headroom matters
- Security auditors require minimal privileged code paths in the main loop
